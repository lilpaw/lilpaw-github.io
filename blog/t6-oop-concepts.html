Classes vs Modules

In Ruby, there are classes and modules. We've been working with classes for a couple of weeks, so I'll first focus on modules.

Modules are tools to bundle objects together, a collection of methods and constants. They contain methods, and can hold classes as well. Modules have two purposes - namespacing and mixin.

Namespacing: a functionality that prevents interference of common names between packages. In other words, it allows classes or modules with conflicting names to co-exist without collisions.

Mixin: allows the sharing of methods across multiple classes or modules. The example we're all familiar with at this point is Enumerable (as you notice, modules are given adjective names). Enumerable methods can be used by any classes that have an .each method.

Classes are "first-class" objects. When a class is created, it is a new object and an instance of the class Class. Classes contain instantiation and methods. They also inherit from modules, but modules do not inherit from classes. Regarding names, classes are given noun names.

Classes create objects and they inherit behavior, and they can also be a base for inheritance. Modules can be inherited, but do not inherit behavior from anything else. Modules can be included in classes and other modules.

Titling this top as a "vs" suggests that we have a choice of one over the other. From my research, I find that this isn't so. Modules and classes work together and knowing module capabilities can help when creating classes.
